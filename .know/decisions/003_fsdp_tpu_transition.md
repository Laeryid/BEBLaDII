# ADR 003: Переход на FSDP для обучения на TPU v6e

## Статус
Предложено

## Контекст
При использовании TPU v6e-4/8 стандартный подход Data Parallel приводит к `RESOURCE_EXHAUSTED` (OOM). 
Модель DeepSeek-R1-7B в BF16 занимает ~14 ГБ HBM, что оставляет всего ~2 ГБ на одном ядре (при общем объеме 16 ГБ) для модели ученика (ModernBERT-Large), активаций и состояний AdamW. 
Накопление градиентов на 8-м шаге вызывает фатальный OOM.

## Решение
Использовать **XlaFullyShardedDataParallel (FSDP)** для распределения весов учителя и ученика между всеми ядрами TPU.

### Детали реализации:
1. **Sharding**: Распределение параметров (Sharding) позволит снизить потребление памяти весами на каждом ядре в N раз (где N - количество ядер, например 8).
2. **Mixed Precision**: Использовать BF16 для всех вычислений (уже используется).
3. **Auto-wrap**: Обернуть слои `Qwen2DecoderLayer` и `ModernBERT` блоки для гранулярного шардинга.
4. **Frozen Teacher**: Даже замороженные веса учителя будут шардированы.

## Последствия
- **Плюсы**: 
    - Возможность обучения с `max_length=4096`.
    - Увеличение эффективного размера батча.
    - Стабильность на TPU v6e-8.
- **Минусы**:
    - Усложнение кода инициализации.
    - Дополнительные накладные расходы на коммуникацию между ядрами (All-Gather параметров во время forward/backward).
